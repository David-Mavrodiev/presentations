ИЗЧИСЛИТЕЛЕН МОДЕЛ ЗА ОПТИМИЗАЦИЯ НА ТЕКСТ:
СЪДЪРЖАТЕЛНО РЕДУЦИРАНЕ ЗА ПОДОБРЯВАНЕ НА СМИСЛОВОТО ВЪЗПРИЯТИЕ НА ИНФОРМАЦИЯТА

  • Дипломант: Давид Мавродиев
  • Научен ръководител: проф. д-р Теодора Бакърджиева
  • Консултант: Salman Khan, PhD
  • Основна цел:
    Създаване на изчислителен модел (алгоритъм) за оптимизация на текст.
  • Условие:
    Моделът да редуцира текст без да се губи съдържателна информация.
  • Следствие:
    Редуцирането да подобри смисловото възприятие на информацията.
  • Главна задача:
    Чрез валидиране на хипотези да се докаже нужда от разработване на модела.
  • Допусната вероятност (хипотези):
      Необходим е изчислителен модел за текстова оптимизация:
    • Решаващ проблеми на читателската аудитория.
    • В помощ на медийната информационна среда.
    • Ако в практиката не съществува подобен модел.
I. КОНЦЕПТУАЛНА РАМКА: ИНФОРМАЦИОННО МОДЕЛИРАНЕ
    В първа част се разглеждат от информационна гледна точка:
  • актуални проблеми (с насока към възможни решения).
  • понятиен апарат: модулност, модалност, модификация.
  • обекти: потребителска аудитория и медийна среда.
  • методология: индекс-индикатор, бинарни опозиции, типологизация, литературни източници.
    Акценти на проблемността:
  • регулация и ентропия на каналите.
  • дефицит на сканиране и филтриране.
  • експанзия и експлозия на инфопотока.
  • разбираемост и възприемане на смисъла.
  • централизация и масовизация на съдържанието.
  • фрагментараност и краткосрочност на посланията.
II. АНАЛИТИЧНА РАМКА: ТЕКСТОВО ПОЗНАНИЕ
    Във втората част се представят от текстова гледна точка:
  • алтернативни решения (в отговор на проблемността).
  • терминология: проекция, презумпция, перцепция.
  • базови модели: скритият на Марков, условни случайни полета, опорни вектори,
    наивният Бейсов- вариации Гаусов, Бернулиев, Полиномен.
  • методи: сравнителен, съпоставителен (benchmarking)
    и SWOT анализи, допитване (анкетно проучване).
    Съвременни решения:
  • извличане на концепции (Concept extraction).
  • информационно извличане (Information extraction).
  • документна класификация (Document classification).
  • извличане на знания от уеб източници (Web mining).
  • обработка на естествен език (Natural Language Processing).
  • документно групиране, т.е. клъстеризация (Document clustering).
  • търсене и извличане на информация (Search and information retrieval).
III. ПРАКТИКУМНА РАМКА: ИЗЧИСЛИТЕЛЕН МОДЕЛ
    В третата част е реализацията на проекта:
  • Същност на оптимизацията (изготвяне на модела).
    • Характеристики:
      етапност, показатели, свойства, параметри, действия, обхват.
  • Подбор на инструментариум (целеви похват).
    • Компоненти:
      токенизация, почистване, разпознаване на части от речта и на именувани същности,
      синтактичен и семантичен анализ.
  • Приложение на алгоритъма (създаване на апликацията “Textopt”).
    • Подход:
      избор на метрики, избор на оценители, разработване на софтуерен продукт.

How Bazel does it?

Definition of terms...

Workspace

Refers to the root directory which contains all source files + external dependencies configuration.
A workspace directory contains WORKSPACE.bazel file which defines the main repository and also describes references to external repositories required for the build.

Repository

Directory containing source code and build instructions
The WORKSPACE directory is the root of the main repository, also called @. Other, external repositories are defined in the WORKSPACE.bazel file. Repository itself could be regarded as a set of packages...

Package

Subdirectory in a repository which has build instructions defined
A package directory includes all files in its directory, plus all subdirectories beneath it and contains BUILD.bazel file which defines targets...

Target

Definition of an action. Most targets are one of two principal kinds:
Rule
A function that has files as input and output.
Chaining between rules is supported. In other words, one rule input files could be another rule output files.
File
Could be source files or generated files as an output from some rule.

Label

Identifier for target

Why so many configurations?

Sample workspace structure
The simple answer is Bazel wants you to split your monolitic build into smaller pieces (targets) as much as possible. But why?

Concepts...

Action Graph

The Action Graph is Direct Acyclic Graph (DAG) representing dependency relations between targets.

Effective Caching

At each incremental build, Bazel analyzes and reconstructs the Action Graph in order to find the minimum subforest of targets it must rebuild.

Parallel Execution

Independent targets could be executed in parallel.

Stages of the Bazel build process

  1. Loading phase
    Load and evaluate all BUILD.bazel files that are needed for the build. The execution of the BUILD.bazel files simply instantiates rules (each time a rule is called, it gets added to a graph).
  2. Analysis phase
    The code of the rules is executed (their implementation function), and actions are instantiated. An action describes how to generate a set of outputs from a set of inputs. In other words, the Analysis phase takes the graph generated by the Loading phase and generates an action graph.
  3. Execution phase
    Actions are executed, when at least one of their outputs is required. If a file is missing or if a command fails to generate one output, the build fails.

Syntax of Bazel

Bazel build files are written in Starlark(dialect of Python).
